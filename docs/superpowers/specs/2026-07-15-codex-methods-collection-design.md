# Codex 方法合集设计

日期：2026-07-15

## 目标

将现有 `/discover` 从通用内容发现页改造成固定的 Codex 方法合集。页面面向公司内部同事，以平铺案例榜呈现经过验证、可以直接复制执行的 Codex 用法，降低用户从公开资料自行学习和转译的成本。

首页继续负责精选推荐、快速匹配和分类内容。通用内容检索保留在首页及分类二级页，不再占用一级导航。

## 范围

本次包含：

- 顶部导航“发现”改为“Codex 方法”
- `/discover` 完全替换为 Codex 方法合集
- 独立静态方法数据，不依赖飞书内容主表
- 首批整理 12 个公开 Codex 用法
- 关键词搜索、场景筛选和提示词复制
- 来源、安全边界、使用条件和最后验证日期
- 每周更新与下线 SOP

本次不包含：

- 自动抓取或自动发布外部案例
- 飞书多维表格字段改造
- 用户账户、收藏、评论或云端点赞
- 学习进度、课程路径或排行榜

## 信息架构

### 顶部导航

- 将“发现”改为“Codex 方法”
- 保留搜索图标，链接到 `/discover?focus=search`
- 方法页加载后，带 `focus=search` 时自动聚焦搜索框

### 页面首屏

首屏保持紧凑，不制作营销型大 Hero，包含：

- 页面标题：Codex 方法合集
- 一句话定位：找到一个工作问题，复制方法，让 Codex 先完成可验证草稿
- 当前收录数量
- 本周更新时间
- 搜索框
- 场景筛选

场景固定为：

1. 电脑维护
2. 资料处理
3. 飞书协作
4. 网页流程
5. 内容生产
6. 开发与测试

### 平铺案例榜

- 桌面端三列
- 平板端两列
- 手机端单列
- 卡片按 `sortWeight` 排序
- 搜索和场景筛选可以组合
- 无匹配结果时提供清除筛选操作

### 页尾说明

- 安全边界
- 收录标准
- 来源与版权说明
- 每周验证机制

## 方法卡片

每张卡片固定展示：

1. 编号
2. 场景分类
3. 解决什么问题
4. 能实现的效果
5. 可直接复制给 Codex 的完整提示词
6. 复制按钮与成功、失败反馈
7. 使用条件：预计耗时、网络要求、风险等级
8. 来源：公开案例作者与能力依据
9. 最后验证日期

卡片不使用内容预览图。视觉重点放在标题、效果和可复制提示词，避免图片挤占信息空间。

卡片文案遵循现有规范：中文结尾不加句号，标题和正文使用固定信息轨道保证横向对齐。

## 数据模型

独立静态文件 `src/data/codex-methods.ts` 导出只读数组。每条记录包含：

```ts
type CodexMethod = {
  id: string;
  number: number;
  category: CodexMethodCategory;
  title: string;
  problem: string;
  outcome: string;
  prompt: string;
  timeToValue: "10 分钟" | "1 小时" | "半天";
  networkRequirement: "无需 VPN" | "部分资源需要 VPN" | "需要 VPN";
  riskLevel: "低风险" | "需确认" | "高风险";
  caseSource?: { label: string; url: string };
  capabilitySource: { label: string; url: string };
  verifiedAt: string;
  sortWeight: number;
  status: "published" | "retired";
};
```

发布数据只读取 `status: "published"` 的记录。`caseSource` 可以缺省，但 `capabilitySource` 必须存在。

## 内容原则

参考公开页面的信息架构和公开事实，但不整页复制原文。首批方法基于公开案例重新组织标题、效果、提示词和安全边界，并保留来源链接。

每条方法必须满足：

- 普通同事能理解它解决什么问题
- 用户能直接复制提示词开始执行
- 涉及删除、付款、发送、发布、权限变更或结束进程时，提示词必须要求 Codex 先说明风险并等待确认
- 至少提供一个官方文档或开源项目作为能力依据
- 公开案例链接无法支撑结论时不得使用
- 不包含公司内部信息、个人隐私或未经脱敏案例

## 首批内容

首批 12 个方法覆盖：

1. 只读诊断网络问题
2. 安全检查磁盘空间
3. 排查 Browser 或 Computer Use 连接
4. 整理资料并生成结构化交付物
5. 通过飞书 Bridge 读取和处理文档
6. 跑网页流程并在关键动作前暂停
7. 生成可玩的网页原型
8. 对本地网页执行真实 QA
9. 生成项目 AGENTS.md
10. 选择并使用 Codex Skills
11. 用目标模式推进长任务
12. 提交前执行代码审查

## 交互行为

### 搜索与筛选

- 搜索字段覆盖标题、问题、效果、提示词、分类和来源标签
- 分类按钮使用 `aria-pressed`
- 搜索结果数量通过 `aria-live` 更新
- URL 可以保存分类和搜索参数，刷新后状态不丢失

### 复制

- 复制按钮只复制提示词正文
- 成功显示“已复制给 Codex”
- 失败显示“复制失败，重新复制”
- 状态变化不得导致按钮尺寸或卡片高度跳动
- 键盘可以完成聚焦和复制

### 来源

- 外部来源在新标签页打开
- 使用 `rel="noreferrer"`
- 区分“公开案例”和“能力依据”

## 每周更新 SOP

每周人工维护一次，不自动发布：

1. 扫描公开 Codex 案例和官方能力更新
2. 判断是否解决明确工作问题
3. 验证来源链接仍可访问
4. 核对 Codex 当前是否具备所述能力
5. 检查提示词是否包含必要确认边界
6. 在隔离环境运行最小验证任务
7. 更新 `verifiedAt`
8. 新内容通过测试后发布
9. 失效内容改为 `retired`，不从历史数据中删除

## 错误与降级

- 静态数据校验失败时构建必须失败
- 单条来源不可访问不影响静态页面打开，但该方法不得继续标记为已验证
- Clipboard API 不可用时给出明确失败反馈，不宣称复制成功
- JavaScript 未加载时，方法内容和来源仍可阅读
- 没有匹配结果时保留搜索框和筛选，不显示空白页面

## 测试与验收

### 单元测试

- 数据模型拒绝未知分类、非法日期、缺少能力依据和未知状态
- 发布列表排除 `retired`
- 搜索覆盖标题、效果、提示词和来源
- 分类与搜索组合过滤正确

### E2E

- 顶部导航显示“Codex 方法”并链接 `/discover`
- `focus=search` 自动聚焦搜索框
- 搜索、分类、清除筛选和结果计数可用
- 复制成功、失败和重试流程可用
- 所有公开来源链接属性正确
- 桌面三列、平板两列、手机单列
- 卡片信息轨道横向对齐且无文本溢出
- 页面无横向滚动，键盘操作可用

### 发布验收

- 首批至少 12 条方法
- 每条包含效果、提示词、能力依据、风险等级和验证日期
- 高风险动作提示词全部包含确认步骤
- 公开内容扫描不包含内部链接、密钥、个人身份和草稿字段

## 成功标准

用户进入页面后，应在 30 秒内完成以下流程：

1. 找到与当前问题匹配的方法
2. 看懂能实现的效果和使用条件
3. 复制提示词给 Codex
4. 查看能力依据和风险边界

页面的价值不在于展示更多链接，而在于把公开资料转译成可直接使用、来源清楚、每周验证的 Codex 工作方法。
