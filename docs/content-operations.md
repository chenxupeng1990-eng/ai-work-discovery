# 内容运营手册

## 发布边界

公开站的唯一权威数据集是 `apps/web/src/generated/content.json`。同步任务只把“发布状态 = 已发布”且“公开级别 = 公开或脱敏案例”的内容写入该文件。草稿、已下架和禁止发布记录不会出现在公开站。

AI 只生成草稿和建议字段，不会自动发布，也不会自动设置公开级别。每条内容必须经过人工审核，并由编辑明确设置为“已发布”和“公开”或“脱敏案例”。

## Base 表结构

字段名必须与下表中文名称完全一致。

| 字段 | Base 类型 | 允许值或格式 | 必填 |

| 生成标题 | 单行文本 | AI 建议，人工可修改 | 否，系统写入 |
| 生成摘要 | 多行文本 | AI 建议，人工可修改 | 否，系统写入 |
| 生成推荐理由 | 多行文本 | AI 建议，人工可修改 | 否，系统写入 |

## 日常流程

1. 编辑在“灵感收件箱”填写“原始内容”，将“处理状态”设为“待处理”。
2. 同步任务把记录标记为“处理中”，检测来源、抓取受限的公开元数据，并请求 AI 生成结构化建议。
3. 系统创建或复用一条“发布状态 = 草稿”的内容记录，创建关联的可复制内容，并把 Inbox 标记为“待审核”。
4. 人工检查事实、链接、版权、敏感信息、图片、标题、摘要、分类、标签和复制块。飞书文档链接必须在无痕窗口验证无需登录即可访问；无法确认时留空，保留有效的原始链接。补齐所有公开发布必填字段。
5. 审核通过后，将内容主表设置为“发布状态 = 已发布”，并根据内容选择“公开级别 = 公开”或“公开级别 = 脱敏案例”。需要时把 Inbox 标记为“已处理”。
6. 下一次同步验证并生成公开 dataset；站点构建后才会展示该内容。

### 下架

将“发布状态”改为“已下架”，或将“公开级别”改为“禁止发布”，然后运行同步。新 dataset 不再包含该记录，静态详情页会在下一次成功部署后消失。需要保留但暂不公开时优先使用“已下架”。

### 手动同步

在 GitHub 仓库打开 **Actions > Sync public content > Run workflow**。workflow 固定检出默认分支，执行同步、单元测试、TypeScript、Astro 检查、构建和公开产物扫描；`verify:public` 通过后才会 stage 和提交 dataset 与内容图片。

本地排障可在 Node 22 环境中运行：

```powershell
cd apps/web
npm ci
$env:FEISHU_APP_ID = "..."
# 其余变量见 docs/deployment.md
npm run sync
npm test
npm run typecheck
npm run check
npm run build
npm run verify:public
```

不要把 secret 作为命令参数，也不要把 `.env`、日志或 Base 原始响应提交到 Git。

## 故障恢复与 last-good

- dataset 先写入临时文件、通过 `PublicDatasetSchema` 后原子替换 `src/generated/content.json`。验证失败时保留上一个有效文件。
- GitHub workflow 在所有验证通过前不会执行 `git add` 或提交；失败运行不会覆盖默认分支上的 last-good dataset。
- 同步失败时先查看 Actions 中固定的错误代码和阶段，修复 Base 字段或 secret 后手动重跑。不要手工提交半成品 JSON。
- 若新内容部署异常，回退或重新部署上一次成功提交；不要删除 last-good dataset。
- 图片下载可能产生未跟踪文件，但 workflow 只在整套验证成功后提交 `public/images/content`。失败运行不会把这些文件推送到仓库。
